# 协同护航模块草案

## 0. 文档目的

这份文档用于把“居间 / 引荐 / 桥接”相关能力，从概念层拉到模块级需求层。

重点不是直接进入开发细节，而是先明确：

- 这个模块到底是什么
- 它为什么必须是一级主菜单
- 它和普通表单、普通发帖区、普通中介介绍页有什么本质区别
- 第一版应该如何设计信息架构、页面结构与流程

## 1. 模块定位

这个模块不能再被简单理解为一句“居间功能”。

它应被定义为：

`多级引荐协作系统 + 资源桥接流程模块 + 带规则与风控保护的链路管理模块`

它的核心任务不是“发一条需求”，而是帮助很多不会操作的居间业务人，通过系统化流程，把整条甲乙双方 + 多级引荐人的链路搭起来。

## 2. 一级主菜单命名候选

这个模块必须放在一级主菜单，不应做成边角功能。

当前推荐名称候选如下：

### 方案 A：`协同护航`

这是当前首选。

优点：

- 字短
- 稳
- 高级
- 不江湖
- 可以同时容纳引荐、桥接、推进、规则、风控

### 方案 B：`链路护航`

优点：

- 更强调“链路”
- 更贴近系统化、过程化、可视化的产品气质

不足：

- 比“协同护航”略偏冷

### 方案 C：`桥接协同`

优点：

- 更直接体现“桥接”
- 能让用户知道这不是普通展示页

不足：

- 读感略硬，没有“协同护航”圆润

当前建议排序：

1. `协同护航`
2. `链路护航`
3. `桥接协同`

补充可参考命名方向：

- `对接链路`
- `资源桥接`
- `引荐协同`
- `协作通路`

如果偏平台感、稳重感，优先保留：

- `协同护航`
- `桥接协同`
- `对接链路`

## 3. 模块核心目标

这个模块的产品目标是七件事：

- 规则引导
- 角色识别
- 链路搭建
- 邀请传递
- 模板辅助
- 风控保护
- 信息分层展示

换句话说，它不是一个发布区，而是一套把“不会操作的居间业务人”带进规范流程的系统。

## 4. 它不是什么

为避免后续跑偏，这里先把几个错误方向排掉。

它不是：

- 普通表单页
- 普通发帖区
- 普通中介介绍页
- 完全公开透明的关系图
- 让所有人都看见全部中间人的系统

## 5. 角色体系

进入模块后，用户不是先填资料，而是先选身份。

当前第一版建议支持以下角色：

- 拍板方 / 决策方
- 一级引荐人
- 二级引荐人
- 其他协同角色

后续如要细分，可进一步拆为：

- 甲方拍板方
- 乙方拍板方
- 甲方一级引荐人
- 乙方一级引荐人
- 二级引荐人
- 法务 / 财务 / 协助角色

## 6. 一级 / 二级引荐人定义

### 一级引荐人

一级引荐人 = 最先接住原始资源、原始需求、原始项目的人，是第一层桥接者。

产品上要特别强调：

- 一级不等于谁声音大
- 一级不等于谁最积极
- 一级是离源头最近、最先形成第一层连接关系的人

### 二级引荐人

二级引荐人 = 在一级引荐人之后继续承接转介绍、再桥接、再撮合的人。

也就是说：

- 他不是源头
- 但他在链路中继续发挥作用
- 他是中间桥段的重要组成部分

## 7. 第一版核心流程

## 第一步：先选身份

用户进入模块后，不直接填表，而是先完成身份选择。

系统先问：

- 你是拍板方还是桥接方
- 你是源头第一承接人，还是继续转介绍的人
- 你是否还需要继续往下连接别人

系统根据选择结果，决定后续页面、字段和提示。

## 第二步：发起一条链路信息

由用户创建一条“链路信息”。

这不是普通帖子，而是一个带关系结构的协作入口。

第一版建议至少包含：

- 基本项目 / 资源描述
- 发起人身份
- 当前已知角色
- 是否需要继续补链
- 当前阶段
- 可附带模板 / 参考协议

## 第三步：系统生成专属链接

发起后，系统自动生成一个专属链接。

用户可以把这个链接发送给：

- 朋友
- 合作方
- 下一层引荐人
- 甲方或乙方相关承接人

## 第四步：逐层补链

收到链接的人进入后，可以在自己权限范围内补充：

- 自己的身份
- 自己处于链条哪一层
- 自己往下要继续连接谁
- 是否补充甲方 / 乙方 / 其他角色信息

## 第五步：系统完成链路搭建

系统最终形成一条完整协作链，内部结构上能明确：

- 谁是甲方
- 谁是乙方
- 谁是拍板方
- 谁是一级引荐人
- 谁是二级引荐人
- 中间还有哪些协同角色

注意：

这是平台内部结构清楚，不等于所有参与者都能看到全部结构。

## 8. 信息展示规则

这个模块必须做“分层展示”，不能全透明。

## 8.1 对居间人 / 引荐人

他们看到的信息应当：

- 公开简明
- 足以理解当前链路和自己任务
- 足以继续往下传递
- 可使用模板和规则

## 8.2 对甲乙双方

不要把所有二级引荐人全部暴露出来。

因为中间层太多时，甲乙双方可能直接失去参与意愿。

因此产品规则必须是：

- 平台系统知道完整链条
- 不同角色只看自己该看的部分
- 甲乙双方不必看到全部中间层细节

## 8.3 第一版权限建议

### 决策方可见

- 项目 / 资源摘要
- 与自己有关的关键节点
- 当前阶段
- 模板与规则说明

默认不看见：

- 所有二级引荐人数
- 全部中间层细节

### 一级引荐人可见

- 自己所在一侧的关键链路
- 当前阶段
- 必要模板
- 上下游关键节点

### 二级引荐人可见

- 简明摘要
- 自己当前所在位置
- 下一步该发给谁
- 可用模板和风险提示

### 平台后台可见

- 完整链条
- 全角色结构
- 时间顺序
- 邀请记录
- 风险提示记录

## 9. 模块内必须带的内容

## 9.1 规则引导

模块内要让很多不会操作的居间业务人清楚知道：

- 自己现在是什么身份
- 该怎么做
- 下一步该发给谁
- 哪些话能说
- 哪些话不能说

## 9.2 风控提示

模块内必须持续体现：

- 边界感
- 保密
- 脱敏
- 合规表达
- 不乱承诺
- 不越权
- 不把自己说成拍板方或结果包办人

## 9.3 合同 / 协议模板

系统里要能挂：

- 参考合同
- 建议模板
- 规则说明

但不是锁死合同。

正确逻辑应是：

- 平台提供建议版
- 甲乙双方可修改
- 中间人可参考，但不应乱改核心条款

## 9.4 形象化表达

这个模块不能做成纯表单堆砌。

需要尽量形象化，让用户一眼看懂：

- 自己在哪一层
- 当前处于哪一步
- 下一步要做什么
- 这条链现在补到哪里了

也就是说，第一版应尽量做成：

- 步骤式
- 节点式
- 链路式
- 带角色标识的可视化结构

## 10. 视觉与交互表达建议

## 10.1 主视觉建议

建议使用一条 `U 型桥接链` 或 `U 型链路` 作为核心图形语言。

含义如下：

- 左侧是甲方
- 右侧是乙方
- 靠近两岸的关键节点，代表一级引荐人
- 中间整段链路，代表二级引荐人与其他协同角色

它的作用不是做特效，而是把抽象规则变成能看懂的关系结构。

## 10.2 动态状态建议

链路应支持逐步点亮：

- 发起时亮起起点
- 新角色加入时补亮一个节点
- 一侧一级引荐形成时，该侧近岸节点高亮
- 甲乙双方都接通时，链路整体闭合或发光

这样用户能直观看到：

- 当前补到哪里
- 还缺哪一段
- 这条链是否已经搭完整

## 11. 第一版信息架构

第一版建议采用 6 个页面 / 区域组成。

这里进一步补充为“文字版页面结构图”，作为后续信息架构和界面设计的直接依据。

## 11.1 页面一：模块首页

目标：

- 解释这是什么模块
- 告诉用户适合谁来用
- 告诉用户不是完全公开关系图
- 引导开始身份选择

页面内容建议：

- 模块一句话说明
- 适用对象
- 合规边界提示
- 进入按钮

文字版页面结构建议：

1. 顶部标题区
- 模块名称
- 一句解释：帮助你把资源、项目、甲乙双方和引荐链路逐步搭起来
- 入口按钮：立即发起一条链路

2. 我的角色入口
- 我是拍板方
- 我是一级引荐人
- 我是二级引荐人
- 我是协同参与者

3. 三步说明区
- 选择身份
- 创建链路
- 邀请补链

4. 风控提醒区
- 不乱承诺
- 不越权
- 先脱敏
- 先确认角色
- 使用平台模板和规则

5. 模板入口区
- 参考合同
- 建议协议
- 操作规则
- 常见问题

## 11.2 页面二：身份识别页

目标：

- 让用户先判断自己属于哪一类角色

页面内容建议：

- 角色选择卡片
- 简短说明
- 系统判断提示
- 进入下一步按钮

文字版页面结构建议：

1. 标题：请选择你在当前链路中的身份
2. 角色卡片
- 拍板方 / 决策方
- 一级引荐人
- 二级引荐人
- 其他协同角色
3. 每个角色下方附带简短中文解释
4. 选择后进入对应流程

## 11.3 页面三：发起链路页

目标：

- 创建一条新的协同链

页面内容建议：

- 项目 / 资源基础描述
- 当前阶段
- 是否需要继续补链
- 可附带模板
- 发起按钮

文字版页面结构建议：

1. 你当前身份
2. 链路标题 / 项目代号
3. 资源 / 项目简述（脱敏版）
4. 当前已知的角色
- 甲方（可选）
- 乙方（可选）
- 当前引荐层级
5. 当前目的
- 搭链
- 找下一层
- 补甲方
- 补乙方
- 补拍板方
6. 是否需要继续邀请他人加入
7. 备注 / 风险提醒

提交后生成：

- 链路编号
- 专属邀请链接
- 当前链路状态

## 11.4 页面四：补链页

目标：

- 让收到邀请的人补充链条中的自己和下一层

页面内容建议：

- 当前链路信息摘要（脱敏）
- 你当前被邀请加入的位置
- 你的身份确认
- 你是否继续向下邀请他人
- 如果继续，填写下一层说明并生成下一个邀请链接
- 当前链路进度展示

## 11.5 页面五：链路详情页

目标：

- 让用户看懂现在搭到了哪里，但不是把所有敏感结构都暴露

页面内容建议：

- 链路标题
- 当前状态
- 当前可见角色节点
- 我的下一步操作按钮
- 风控提示
- 模板 / 合同入口
- 操作记录

展示原则：

- 展示必须分权限
- 平台知道完整结构
- 用户只看到自己该看的部分

## 11.6 页面六：模板与规则中心

目标：

- 给不会操作的人一个稳定抓手

页面内容建议：

- 参考合同
- 推荐模板
- 规则说明
- 合规提示
- 常见错误

至少包含：

1. 一级引荐人说明
2. 二级引荐人说明
3. 拍板方说明
4. 脱敏规则
5. 保密提醒
6. 合规提醒
7. 参考合同
8. 建议模板
9. 常见错误示例

## 11.7 MVP 交付范围建议

第一版不要贪大，建议先交付以下 5 个页面：

1. 模块首页
2. 发起链路页
3. 角色识别页
4. 链路详情页（简版）
5. 模板 / 规则说明页

后续再扩展：

- 更复杂的多级补链
- 更精细的权限控制
- 电子协议协同编辑
- 状态流转和分润机制

## 12. 第一版流程设计草案

第一版可以用下面这条主流程：

1. 进入 `协同护航`
2. 看到模块说明与边界提示
3. 开始身份识别
4. 选择自己是拍板方 / 一级 / 二级 / 其他
5. 创建一条链路信息
6. 系统生成专属链接
7. 发给下一位承接人
8. 对方进入后确认身份并补链
9. 链路工作台实时更新
10. 持续邀请，直到甲乙双方链条逐步补齐
11. 必要时使用模板、规则和风险提示
12. 由平台在后台掌握完整结构，但前台按角色分层展示

## 12.1 低保真页面草图思路

这一模块第一版不需要复杂视觉堆砌，但必须足够“流程感”和“链路感”。

建议低保真结构按以下方式理解：

### A. 模块首页草图思路

- 顶部大标题 + 一句解释
- 中间四张角色入口卡片
- 下方三步流程条
- 再下方风控提醒卡
- 底部模板入口卡组

### B. 发起链路页草图思路

- 左侧或顶部显示“当前身份”
- 中间主表单填写链路标题、脱敏描述、当前目的
- 下方显示“已知角色”卡片区
- 提交后跳到链路详情并生成链接

### C. 角色识别页草图思路

- 页面中央四张角色卡
- 每张卡都有一句“你适合哪种情况”的解释
- 选中后出现下一步按钮

### D. 链路详情页草图思路

- 顶部：链路标题 + 当前状态
- 中间：简化 U 型链路图或节点链路图
- 右侧或下方：我的下一步操作
- 再下方：模板入口、风控提示、操作记录

### E. 模板与规则中心草图思路

- 左侧规则分类导航
- 右侧内容区
- 顶部高亮“你当前身份适合先看什么”

## 12.2 可视化表达补充建议

这块不要只做列表和表单。

界面建议尽量包含：

- 节点卡片
- 层级标识
- 当前步骤高亮
- 链路进度条
- 角色徽标
- “下一步该做什么”的明确引导

视觉目标是让人一眼就明白：

- 我是谁
- 我在哪一层
- 现在走到哪一步
- 下一步要发给谁

## 13. 当前阶段结论

这个模块的产品本质不是一个发帖区，而是：

- 多级引荐协作系统
- 居间链路搭建系统
- 资源桥接流程模块
- 带规则和风控保护的链路管理模块

所以接下来开发时必须坚持四条底线：

1. 不简化成普通表单
2. 不简化成普通中介介绍页
3. 不做成完全公开透明的关系图
4. 必须体现层级、链路、角色、规则、模板、风控和分层可见
